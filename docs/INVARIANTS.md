# DelayKit Invariants

These invariants define the correctness model. Every design choice must preserve them. When a test fails, assume the code is wrong first — only weaken the invariant after verifying the invariant itself is incorrect.

A line belongs here if removing it would change what "correct" means for the library. Feature specs, backoff values, implementation details, and design rationale belong in README or PR descriptions.

**Core principle: the store row is the source of truth. Scheduler wakes are disposable.**

---

## 1. The store is the source of truth

For any key, the store row determines:
- Whether an action is still active
- What version is current
- When it should fire
- Whether it has already completed

The scheduler delivers wake-up signals but never decides which version is current — that is always determined by the store row.

## 2. Every (handler, key) pair has at most one active row

The store enforces: for any `(handler, key)` pair, at most one row is active (pending or running) at any time. This is enforced by a unique partial index on `(handler, key) WHERE status IN ('pending', 'running')`.

Different handlers can use the same key concurrently — `schedule("send-reminder", { key: "user_123" })` and `schedule("expire-trial", { key: "user_123" })` coexist. The handler name provides the namespace; the key is the entity identifier.

The `kind` field determines how the row behaves:
- `once` — fires at `scheduledFor`, runs the handler, reaches terminal state
- `debounce` — accumulates events, fires after activity settles (trailing edge)
- `throttle` — accumulates events, fires once per fixed window (trailing edge)

Within a single handler, a key cannot be owned by both a `once` job and a pattern simultaneously. `schedule()` and `debounce()`/`throttle()` validate `kind` on existing rows.

## 3. Identity model

- **`key`** — business identity. The entity identifier the developer chooses (`user_123`). At most one active row per `(handler, key)` pair.
- **`id`** — row identity. Stable UUID, pre-generated by the caller for scheduler-first flow.
- **`version`** — latest intent. Incremented by user events (`debounce()`/`throttle()` calls) and internal operations (reschedule, requeue).
- **`claimedVersion`** — the version at the time execution was claimed (`markRunning`). Null when not running. `version > claimedVersion` means new events arrived during execution.
- **`schedulerRef`** — the ID of the currently active external scheduler artifact (e.g., PostHook hook ID). Updated on every wake materialization. Used to identify stale deliveries.
- **`deferAttempts` / `deferredSince`** — defer-loop state when the handler isn't registered at delivery (see *Handler-not-registered deferral*). Independent of `attempt` / `maxAttempts`. Reset on `markRunning` and `replaceJob` — every other path into `pending` passes through `markRunning` first.
- **`retryConfig`** — snapshot of the handler's retry config at schedule time, read by the defer path when the handler isn't loaded on the current instance.
- **Wake payload** — a wake carries `{ jobId }` (and optionally `key` for debugging). It says "look at this job," not "execute this specific version." The row decides validity. The key in the payload is not used for correctness.

## 4. Delivery validation

A wake-up is a signal to re-read the row. The row decides whether to execute.

When a wake arrives carrying `(jobId, hookId)`:

1. Load the row by `id`
2. If status is terminal or row missing → ignore (200)
3. **Inline stalled reclaim**: if the row is `running` with an expired lease (`startedAt + timeout + grace < now`), reclaim it back to `pending` before proceeding. This handles redelivery to a job whose previous executor crashed.
4. If status is not `pending` after reclaim check → ignore
5. **Artifact identity guard**: if `schedulerRef` is set and `hookId !== schedulerRef` → stale artifact, ignore (200). The current hook should still exist.
6. **Timing guard**: for `once` jobs, if `scheduledFor > now + 5s` → early delivery, return retry (500) so the scheduler redelivers later. The 5-second grace absorbs clock drift. Returning 200 would strand the job.
7. For debounce: check settlement (`now - lastAt >= waitMs` or maxWait exceeded). If not settled → reschedule and materialize a new wake.
8. Atomic claim: `UPDATE SET status = 'running', claimedVersion = version WHERE id = $1 AND version = $current AND status = 'pending'`
9. If claim fails → another worker claimed it, or state changed. Ignore.

The two-layer guard:
- **schedulerRef** answers "is this the currently active external artifact?" — catches stale hooks from replace, reschedule, and requeue.
- **scheduledFor** answers "is it time yet?" — catches early delivery of the current artifact.

PollingScheduler does not use webhook delivery, so the artifact identity guard and timing guard do not apply — the poller reads the current row directly and uses its version for the claim CAS.

## 5. Cancellation is logical, not physical

Cancelling marks the row as cancelled in the store. Physical cancellation of a scheduler artifact (e.g., deleting the PostHook hook) is best-effort. Stale wakes are safely ignored by invariant #4.

`unschedule(handler, key)` cancels pending rows only. Running rows cannot be cancelled.

## 6. Handlers should be idempotent

Duplicate wakes happen. The store ensures only one execution path wins for a given `(id, version)` via the atomic claim in invariant #4.

The edge case: a handler runs, produces side effects, but crashes before terminal transition. On recovery, `markRunning` succeeds again and the handler re-executes.

## 7. All state transitions go through dedicated mutation methods

The Store interface has no generic partial-update method. Every mutation is purpose-built with explicit preconditions:

- **CAS-guarded transitions**: `markRunning`, `markCompleted`, `markFailed`, `retryJob`, `rescheduleDueAt`, `requeueForNextWindow`, `replaceJob`, `deferJob` — each checks version and/or status before writing.
- **Targeted field writes**: `cancelJob(id)`, `updateScheduledFor(id, scheduledFor)`, `updateSchedulerRef(id, version, ref)` — narrow scope, no arbitrary field access.

This prevents unchecked overwrites. A generic `updateJob(id, Partial<Job>)` was removed after it caused a duplicate-delivery race where a stale path overwrote a newer `schedulerRef`.

## 8. Lifecycle events are observational only

Events (`job:scheduled`, `job:started`, `job:completed`, etc.) are notifications for logging, metrics, and monitoring. They must never participate in correctness:

- Listener errors (sync throws and async rejections) are caught and logged — they never propagate to the job execution path.
- A listener cannot alter a state transition. It cannot change the outcome of a job.
- If all listeners are removed, the system behaves identically.

Listeners are invoked synchronously inline before the next step proceeds. A slow synchronous listener adds latency to the execution path. Keep listeners fast (logging, metrics).

## 9. Wake materialization

### Scheduler-first flow (new jobs)

`schedule()`, `debounce()`, and `throttle()` for new windows:
1. Pre-generate UUID
2. Call `scheduler.schedule()` to create the external artifact
3. Insert the row with `schedulerRef` pointing to the artifact

If step 2 fails, nothing is in the DB. If step 3 fails, the artifact is orphaned but harmless — it will deliver to a nonexistent row and be ignored.

### DB-first flow (reschedule and requeue)

`rescheduleDueAt()` and `requeueForNextWindow()` update the row first, then materialize a new wake:
1. Atomic row transition (CAS on version or status)
2. Call `scheduler.schedule()` to create a new artifact
3. Store the ref via `updateSchedulerRef(id, version, ref)` (version-guarded)

If step 3 fails (version advanced during the round-trip), the orphaned artifact is cancelled. This prevents lost wakeups where the row points at a stale ref and the real hook gets rejected by the identity guard.

### Existing-window events

`debounce()`/`throttle()` calls on a pending row just bump version + lastAt. No new wake is needed — the existing hook will fire at `scheduledFor`, and the executor re-evaluates the row at delivery time.

## The schedule() model

### schedule(handler, { key, delay | at }) — default: onDuplicate = 'skip'

1. Check for existing active row by key
2. If existing row has `kind !== 'once'` → reject (pattern collision)
3. If existing and `onDuplicate = 'skip'` → return existing
4. If no existing → scheduler-first: pre-generate UUID, schedule wake, insert row
5. If concurrent insert conflict → retry with full validation

### schedule(handler, { key, delay | at, onDuplicate: 'replace' })

Same as above, but if existing is pending:
- Atomic UPDATE: increment version, update scheduledFor, reset attempt
- Materialize new wake (sets `schedulerRef` to new hook)
- Cancel old scheduler artifact (best-effort — stale delivery rejected by identity guard)

Running → replace is blocked. The handler owns the row until terminal.

### debounce(handler, { key, wait, maxWait? })

1. Try `updatePatternEvent(key)` — bumps version + lastAt on existing row
2. If no existing row → scheduler-first: pre-generate UUID, schedule wake, insert row
3. If concurrent insert conflict → retry as update on the winner

### throttle(handler, { key, wait })

Same flow as debounce. No maxWait.

### cancel(id) / unschedule(handler, key)

1. Load row (by id or key)
2. If pending → mark cancelled, attempt physical cancellation (best-effort)
3. If running → reject

## State machine

```
pending → running → completed       (once: normal execution)
pending → running → failed          (once: retries exhausted)
pending → running → pending         (once: retry, increment attempt)
pending → cancelled                 (logical cancellation)
pending → pending                   (once: replace, increment version)
pending → pending                   (handler not registered: defer with exponential backoff)
pending → failed                    (handler not registered for longer than the defer horizon)

pending → running → completed       (pattern: no new events during execution)
pending → running → pending         (pattern: new events during execution → requeue)
pending → running → pending         (pattern: retry, increment attempt)
pending → running → failed          (pattern: retries exhausted, no new events)
pending → pending                   (pattern: new event bumps version + lastAt)
```

States:
- **pending** — intent recorded, awaiting wake-up
- **running** — handler executing (claimed by one worker)
- **completed** — handler succeeded, terminal
- **failed** — handler exhausted all retries, terminal
- **cancelled** — logically cancelled, terminal

### Pattern terminal transition

After a pattern handler completes (success or exhausted failure):

1. Try `markCompleted(id, version)` or `markFailed(id, version, error)`
   - Uses `version` from `result.job` (captured at load time, before handler ran)
   - If version hasn't changed → succeeds (no new events during execution)
2. If mark fails (version advanced during execution) → `requeueForNextWindow(id)`
   - Transitions running → pending with new scheduledFor computed from row fields
   - Materializes a new wake afterward

The `claimedVersion` field makes this explicit: `version > claimedVersion` means new events arrived during execution.

### Pattern event during execution

When `debounce()`/`throttle()` is called while the handler is running:
- `updatePatternEvent` bumps version and updates lastAt
- `firstAt` is reset only on the FIRST event after execution started (preserves throttle window anchoring and debounce maxWait stability)
- No new wake is scheduled — the terminal transition handles it via requeue

### Retry ownership

- **PollingScheduler**: DelayKit owns retries. Handler fails → running → pending (retry) or running → failed (exhausted). Retry timing configured on `dk.handle()`.
- **PosthookScheduler**: PostHook owns retry timing. Handler failure returns 500, PostHook retries delivery. DelayKit transitions the row to pending between attempts so the retry delivery can claim it.

### Handler-not-registered deferral

When a wake's handler isn't registered on the current process, the executor returns `handler_not_registered` without mutating the row. The result handler then calls `deferJob`, which transitions the row via one of two branches:

- **Defer** (`pending → pending`): advances `scheduledFor` by an exponential backoff, increments `deferAttempts`, sets `deferredSince` on first defer. `attempt` is not touched — a defer does not consume the user's retry budget.
- **Horizon flip** (`pending → failed`): if `now - deferredSince` has crossed the defer horizon, the row flips terminal with an operator-actionable `lastError`.

`deferAttempts` and `deferredSince` reset on `markRunning` (successful claim) and `replaceJob` (user-requested new schedule). Every other path into `pending` passes through `markRunning` first, so no other reset sites are required.

### Retry + version advance

If a handler fails and `retryJob(id, version, ...)` is attempted:
- If version hasn't changed → retry succeeds (running → pending with new attempt)
- If version advanced → retry CAS fails → `requeueForNextWindow` starts a fresh window instead

## Schema

```sql
CREATE TABLE delaykit.jobs (
  id              UUID PRIMARY KEY,
  kind            TEXT NOT NULL DEFAULT 'once',
  handler         TEXT NOT NULL,
  key             TEXT NOT NULL,
  version         INT NOT NULL DEFAULT 1,
  claimed_version INT,
  status          TEXT NOT NULL DEFAULT 'pending',
  scheduled_for   TIMESTAMPTZ NOT NULL,
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  attempt         INT NOT NULL DEFAULT 0,
  max_attempts    INT NOT NULL DEFAULT 1,
  scheduler_ref   TEXT,
  last_error      TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  first_at        TIMESTAMPTZ,
  last_at         TIMESTAMPTZ,
  wait_ms         INT,
  max_wait_ms     INT
);
```

Indexes:
- `UNIQUE (handler, key) WHERE status IN ('pending', 'running')` — enforces at-most-one-active invariant per handler
- `(scheduled_for) WHERE status = 'pending'` — for polling due jobs
- `(started_at) WHERE status = 'running'` — for stalled job recovery

## Pattern timing

### Debounce (trailing edge)

- `scheduledFor` = time of first event + waitMs (for initial window)
- Wake evaluates settlement: `now - lastAt >= waitMs`
- If not settled: reschedule to `lastAt + waitMs` (capped by `firstAt + maxWaitMs`)
- If settled or maxWait exceeded: fire

### Throttle (trailing edge)

- `scheduledFor` = time of first event + waitMs (fixed window)
- Wake always fires at end of window — no settlement check
- Requeue uses `firstAt + waitMs` (window anchored to first event of burst)

### firstAt semantics during execution

When events arrive while a handler is running:
- First event resets `firstAt` to now (start of next burst)
- Subsequent events update only `lastAt`
- This ensures throttle windows and debounce maxWait are anchored to the first event of the new burst, not the last

## Race conditions and recovery

### Stale hook after replace

When `schedule(..., onDuplicate: "replace")` moves a job later:
1. `replaceJob` increments version, updates `scheduledFor`, clears `schedulerRef`
2. `materializeWakeup` creates a new hook and stores its ref as `schedulerRef`
3. Old hook is cancelled (best-effort)

If the old hook still delivers: `hookId !== schedulerRef` → rejected by the artifact identity guard.

### Stale hook after debounce reschedule

When a debounce hook fires but the pattern is not settled:
1. `rescheduleDueAt` updates `scheduledFor` and increments version
2. `materializeWake` creates a new hook and stores the ref via version-guarded CAS

If the old hook redelivers: `hookId !== schedulerRef` → rejected.

### Orphaned hook on failed CAS

If the row version advances during the `schedule()` round-trip (e.g., a new pattern event arrives), `updateSchedulerRef` CAS fails. The newly created hook is orphaned — the row doesn't know about it. If left live, it would deliver and be rejected by the identity guard, stranding the current window.

Fix: on CAS failure, cancel the orphaned hook immediately.

### Concurrent debounce/throttle calls

Two concurrent `debounce()` calls for the same key:
- `updatePatternEvent` uses an atomic UPDATE with WHERE guards. One wins, bumping version. The other either succeeds too (both bump independently) or finds no row and falls through to insert. Concurrent inserts are caught by the unique constraint — the loser retries as an update.

### Concurrent schedule() calls

Two concurrent `schedule()` calls for the same key:
- First: insert succeeds
- Second: insert fails (concurrent insert), retries with full validation — finds the winner, applies skip/replace/handler-mismatch rules

### Stalled running recovery

If a worker dies mid-execution, the row stays in running with a stale `startedAt`. Recovery happens through two paths:

1. **Bulk sweep** (`reclaimStalledJobs`): Called by PollingScheduler on a timer and by `poll()` at the start of each cycle.
2. **Inline reclaim** (`reclaimStalled`): Called by `executeJob` when a wake arrives for a running job with an expired lease.

In both cases:
- If `attempt + 1 < maxAttempts` → transition back to pending
- If attempts exhausted → transition to failed
- For patterns with version advance (events during stalled execution) → requeue fresh window
